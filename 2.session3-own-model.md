## What "Your Own Model" Really Means

You don't train a model from scratch (that costs millions of dollars and months of compute). What you do is:

1. Use an existing model (Gemini, DeepSeek, Qwen, MiniMax) via API
2. Control it completely with your system prompt, temperature, and parameters
3. Wrap it in your own app — CLI tool, web app, bot, automation script
4. Optionally fine-tune it on your own data (advanced — see Step 10)

The API key is your access pass to the model. The system prompt is how you make it *yours*.

```
Your App  →  OPENROUTER_API_KEY  →  OpenRouter  →  google/gemini-3.5-flash
                                               →  deepseek/deepseek-v4-pro
                                               →  qwen/qwen3.7-max
                                               →  minimax/minimax-m3
```

## Step 3 — Your First Real API Call

OpenRouter uses the exact same SDK as OpenAI. The only change is `base_url`.

```python
import os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()

client = OpenAI(
    api_key=os.getenv("OPENROUTER_API_KEY"),
    base_url="https://openrouter.ai/api/v1"    # ← this is the only difference
)

response = client.chat.completions.create(
    model="google/gemini-3.5-flash",
    messages=[
        {"role": "system", "content": "You are a helpful DevOps engineer."},
        {"role": "user",   "content": "Explain what a Kubernetes namespace is."}
    ]
)

print(response.choices[0].message.content)
```

**What the response object contains:**
```python
response.choices[0].message.content   # the reply text
response.model                         # model that answered
response.usage.prompt_tokens           # tokens you sent
response.usage.completion_tokens       # tokens in the reply
response.usage.total_tokens            # total billed tokens
```

## Step 4 — Understanding Parameters

### 4.1 — `model`

Switch models by changing one string. Each has different strengths:

| # | Model | Best For |
|---|-------|----------|
| 1 | `google/gemini-3.5-flash` | Fast tasks, writing, basic Q&A |
| 2 | `minimax/minimax-m3` | Multi-turn conversations, chatbots |
| 3 | `qwen/qwen3.7-max` | Code generation, commit messages |
| 4 | `deepseek/deepseek-v4-pro` | Reasoning, log analysis, code review |

### 4.2 — `max_tokens`

Limit the length of the response (controls cost and speed):
- Simple answer: 200–500
- Code snippet: 500–1000
- Full document or runbook: 2000–4000

```python
response = client.chat.completions.create(
    model="google/gemini-3.5-flash",
    max_tokens=500,
    messages=[...]
)
```

### 4.3 — `temperature`

```
0.0  → Deterministic. Same answer every time. Use for code, extraction.
0.3  → Slightly varied. Use for structured writing, summaries.
0.7  → Creative. Use for brainstorming, conversation.
1.0+ → Wild. Use for creative fiction, lateral thinking.
```

### 4.4 — `system` message (System Prompt)

The system message is where your "model personality" lives — permanent instructions for every response.

```python
# Weak — too vague
{"role": "system", "content": "Be helpful."}

# Strong — RBCFC style
{"role": "system", "content": """You are a senior Azure DevOps engineer.
Write concise, production-ready answers.
Always include CLI commands.
Never suggest solutions that require portal access.
Format code in code blocks."""}
```

## Step 5 — Multi-Turn Conversations (Memory)

Models have no memory between API calls — you pass the full history every time.

```python
import os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()
client = OpenAI(
    api_key=os.getenv("OPENROUTER_API_KEY"),
    base_url="https://openrouter.ai/api/v1"
)

SYSTEM = "You are a helpful DevOps assistant."

def chat(history: list, user_message: str) -> str:
    history.append({"role": "user", "content": user_message})

    response = client.chat.completions.create(
        model="minimax/minimax-m3",
        messages=[{"role": "system", "content": SYSTEM}] + history
    )

    reply = response.choices[0].message.content
    history.append({"role": "assistant", "content": reply})
    return reply

# Model remembers context across turns
history = []
reply, history = chat(history, "I'm setting up AKS."), history
print(reply)

reply = chat(history, "What node size should I use for general workloads?")
print(reply)
```

See `applications/07_terminal_chatbot.py` for the full interactive version.

## Step 6 — Structured Output (JSON from the Model)

Make the model return JSON you can parse and use directly in code.

```python
import json, os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()
client = OpenAI(
    api_key=os.getenv("OPENROUTER_API_KEY"),
    base_url="https://openrouter.ai/api/v1"
)

def get_structured_output(prompt: str) -> dict:
    system = """You are a data extractor.
Always respond with valid JSON only.
No markdown, no explanation, no backticks. Just the JSON object."""

    response = client.chat.completions.create(
        model="deepseek/deepseek-v4-pro",
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": prompt}
        ]
    )
    return json.loads(response.choices[0].message.content)

# Example: extract structured data from a Kubernetes error
result = get_structured_output("""
Extract from: "0/3 nodes available: 3 Insufficient cpu."
Return JSON: {"error_type": "", "affected_resource": "", "root_cause": "", "fix": ""}
""")
print(result["fix"])
```

## Step 7 — Streaming Responses

For long outputs, stream tokens as they arrive — much better UX than waiting for the full response.

```python
with client.chat.completions.stream(
    model="deepseek/deepseek-v4-pro",
    messages=[{"role": "user", "content": "Write a detailed runbook for AKS upgrades."}]
) as stream:
    for chunk in stream:
        if chunk.choices[0].delta.content:
            print(chunk.choices[0].delta.content, end="", flush=True)
```

## Step 8 — Build a Real Tool: PR Description Generator

A complete, runnable DevOps tool. Pipe a git diff in, get a professional PR description out.

```python
"""
Usage: git diff main...HEAD | python pr_description.py
"""
import os, sys
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()
client = OpenAI(
    api_key=os.getenv("OPENROUTER_API_KEY"),
    base_url="https://openrouter.ai/api/v1"
)

SYSTEM = """You are a senior engineer writing GitHub pull request descriptions.
Given a git diff, write a PR description with:
1. What changed — 2-3 sentence summary
2. Why — business or technical reason
3. Type — feat / fix / refactor / chore / docs
4. Testing — what to verify
5. Checklist — [ ] tests pass  [ ] docs updated  [ ] no secrets

Use markdown. Be concise."""

def generate_pr_description(diff: str) -> str:
    response = client.chat.completions.create(
        model="qwen/qwen3.7-max",
        messages=[
            {"role": "system", "content": SYSTEM},
            {"role": "user",   "content": f"Git diff:\n\n{diff}"}
        ]
    )
    return response.choices[0].message.content

if __name__ == "__main__":
    diff = sys.stdin.read()
    if not diff.strip():
        print("Usage: git diff main...HEAD | python pr_description.py")
    else:
        print(generate_pr_description(diff))
```

## Step 10 — Fine-Tuning (When and Why)

Fine-tuning = training an existing model on your own data so it specializes.

| # | Approach | What you do | Cost | When to use |
|---|----------|------------|------|-------------|
| 1 | Prompting | Write a strong system prompt | Free | 90% of cases — start here |
| 2 | Few-shot | Add examples inside the prompt | Free | When prompting alone isn't consistent |
| 3 | Fine-tuning | Train on your own dataset | $$$ | Specific domain, very consistent style needed |
| 4 | RAG | Give model access to your docs at query time | Low | Large or frequently updated knowledge base |

**Rule of thumb:** A well-crafted RBCFC system prompt covers 90% of use cases. Fine-tuning is rarely needed at the start.

---

## Step 11 — RAG (Retrieval Augmented Generation)

RAG = give the model access to your documents at query time, without retraining it.

```
1. User asks a question
         ↓
2. Search your document store (vector database)
         ↓
3. Retrieve the most relevant document chunks
         ↓
4. Inject into prompt:
   "Relevant docs: [chunks]. Now answer: [question]"
         ↓
5. Model answers using your actual docs as context
```

**DevOps use cases:**
1. RAG on internal runbooks → AI-assisted incident response
2. RAG on Confluence pages → instant internal knowledge search
3. RAG on past incident reports → pattern detection and repeat prevention

**Tools to explore:** LangChain, LlamaIndex, Pinecone, Azure AI Search
